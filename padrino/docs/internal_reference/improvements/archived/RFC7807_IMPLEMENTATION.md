# RFC 7807: Problem Details for HTTP APIs

This document outlines the implementation strategy for [RFC 7807](https://tools.ietf.org/html/rfc7807) within the **Don
** project.

## Overview

RFC 7807 provides a standardized format for reporting errors from HTTP APIs. This ensures that clients receive
consistent, machine-readable information regardless of the specific error type.

## Response Structure

The standard response uses the `application/problem+json` media type and contains the following fields:

| Field      | Type   | Description                                               | Mapping from `AppError`             |
|:-----------|:-------|:----------------------------------------------------------|:------------------------------------|
| `type`     | URI    | Identifies the problem type. Defaults to `about:blank`.   | `AppError.Type` (or generated)      |
| `title`    | String | A short, human-readable summary.                          | `AppError.Message`                  |
| `status`   | Number | The HTTP status code generated by the server.             | `AppError.HttpStatus`               |
| `detail`   | String | A human-readable explanation specific to this occurrence. | `AppError.Detail.Detail`            |
| `instance` | URI    | A URI reference that identifies the specific occurrence.  | `echo.Context.Request().RequestURI` |
| `code`     | String | **Extension:** Machine-readable application error code.   | `AppError.Code`                     |

## Mapping Strategy

### 1. Domain Layer (`internal/core/domain`)

Business logic returns pure domain errors. These errors are defined using the `AppError` struct, which includes an HTTP
status code and a unique machine-readable code (e.g., `EDON0001`).

```go
// Example Domain Error
ErrNotFound = &apperr.AppError{
Code:       "EDON0001",
Message:    "The requested resource was not found",
HttpStatus: 404,
}
```

### 2. Adapter Layer (`pkg/server/echoserver`)

The `DefaultErrorHandler` acts as the translator. It intercepts any error returned by a handler and transforms it into
the `ProblemDetails` JSON structure.

#### Transformation Logic:

1. **Identify:** Check if the error is an `*apperr.AppError`.
2. **Contextualize:** Extract the `RequestURI` from the Echo context to populate the `instance` field.
3. **Map:** Assign domain fields to RFC fields.
4. **Fallback:** If the error is unknown, return a `500 Internal Server Error` with `about:blank` as the type.

## Implementation Examples

### Problem Details Struct
This struct resides in the transport layer (e.g., `pkg/server/echoserver/error_handler.go`) and defines the JSON schema for the response.

```go
type ProblemDetails struct {
	Type     string `json:"type"`
	Title    string `json:"title"`
	Status   int    `json:"status"`
	Detail   string `json:"detail,omitempty"`
	Instance string `json:"instance,omitempty"`
	Code     string `json:"code,omitempty"` // Extension
}
```

### Centralized Error Handler
The implementation of the `DefaultErrorHandler` in Echo that performs the mapping:

```go
func DefaultErrorHandler(err error, c echo.Context) {
	if c.Response().Committed {
		return
	}

	status := http.StatusInternalServerError
	title := "Internal Server Error"
	code := "INTERNAL_ERROR"
	var detail string

	var appErr *apperr.AppError
	if apperr.As(err, &appErr) {
		status = appErr.HttpStatus
		title = appErr.Message
		code = appErr.Code
		detail = appErr.Detail.Detail
	}

	resp := ProblemDetails{
		Type:     "about:blank", // Or a specific documentation link
		Title:    title,
		Status:   status,
		Detail:   detail,
		Instance: c.Request().RequestURI,
		Code:     code,
	}

	c.JSON(status, resp)
}
```

### Usage in Handlers
Handlers only need to return domain errors. The infrastructure takes care of the rest.

```go
func (h *UserHandler) Get(c echo.Context) error {
    user, err := h.usecase.Find(c.Request().Context(), c.Param("id"))
    if err != nil {
        // Just return the error; the ErrorHandler will format it
        return err 
    }
    return c.JSON(http.StatusOK, user)
}
```

## Example Transformation

### Input (Domain Error)

```go
domain.ErrConflict.WithDetail(apperr.ErrorDetail{
Detail: "User with email 'igor@example.com' already exists",
})
```

### Output (HTTP Response)

```json
{
  "type": "about:blank",
  "title": "There is a conflict with the current state of the resource",
  "status": 409,
  "detail": "User with email 'igor@example.com' already exists",
  "instance": "/v1/users",
  "code": "EDON0006"
}
```

## Benefits

- **Standardization:** All errors follow a predictable format.
- **Interoperability:** Client libraries designed for RFC 7807 can automatically parse these errors.
- **Traceability:** The `instance` field helps correlate client-side errors with server-side logs.
- **Extensibility:** The `code` field preserves our internal error catalog for legacy or specific machine-level
  handling.
